# Adaptive Prompt System — Cross-Project Plan

## Problem

The brain routing prompt is hardcoded in `brain/prompt.py` — a single system prompt
that must work for every model from 1B to 70B+. This creates two failures:

1. **Small models** (Qwen2.5-7B) drown in long, nuanced instructions. They latch onto
   keywords ("conversational" → awareness) instead of reasoning about intent.
   "can you tell me a funny joke" routes to awareness; "tell me a joke" synthesizes a nerve.

2. **Large models** (32B+) get under-utilized — they could handle richer context,
   few-shot examples, and subtle routing logic, but receive the same minimal prompt.

The same problem applies to all roles: brain, nerve, creative, communication, awareness.

## Design

### Core Principle

Every role has a **core prompt** — the minimal, TinyML-grade instruction that works on
the smallest model. This is the atomic unit of the role's purpose. It must be:
- Short (under 500 tokens)
- Explicit (no ambiguity, no "conversational exchanges")
- Structured (numbered rules, not prose)
- Universal (works on 1B–70B)

### Prompt Hierarchy

```
sentient-community/adapters/brain/{model_name}/
  ├── meta.json            # model, size_class, provider, capabilities
  ├── context.json         # system_prompt, few_shot_examples, temperature
  └── qualification.json   # score, test results (generated by critic)

sentient-community/adapters/brain/core/
  └── context.json         # THE core prompt — TinyML baseline, protected
```

Same structure for each role:
```
adapters/{role}/core/context.json        # Core prompt for role
adapters/{role}/{model_name}/context.json # Model-specific override
```

Roles: `brain`, `nerve`, `creative`, `code`, `awareness`, `communication`, `vision`

### Prompt Resolution Order

```
1. adapters/{role}/{exact_model}/context.json     # Exact model match
2. adapters/{role}/{size_class}/context.json       # Size class fallback (tiny/small/medium/large)
3. adapters/{role}/core/context.json               # Core universal prompt
```

### Schema (already exists in community)

`schemas/adapter_context.schema.json`:
```json
{
  "system_prompt": "string (required)",
  "few_shot_examples": [{"input": "...", "output": "..."}],
  "temperature": 0.7,
  "top_p": 0.9,
  "max_tokens": 512,
  "stop_sequences": []
}
```

`schemas/adapter_meta.schema.json`:
```json
{
  "model": "Qwen2.5-Coder-7B",
  "size_class": "small",         // tiny (<3B), small (3-13B), medium (13-34B), large (34B+)
  "provider": "gguf",
  "contributor": {"github": "..."},
  "capabilities": {
    "json_mode": true,
    "tool_calling": false,
    "max_context": 4096
  }
}
```

## Implementation Plan

### Phase 1: Core Prompts (sentient-community)

**Goal**: Extract the current hardcoded prompts into core adapter files.

#### 1.1 Create core brain prompt
- File: `adapters/brain/core/context.json`
- Rewrite `brain/prompt.py` system prompt as a minimal, directive prompt
- Key fix: Remove "ALL conversational exchanges" from awareness description
- Key fix: Priority order = existing nerves first → senses → synthesize
- Key fix: Remove contradictory "respond directly" instruction (line 105)
- Add 3-5 few-shot examples for common routing patterns (joke → nerve, hello → awareness)

#### 1.2 Create core nerve planner prompts
- `adapters/nerve/core/context.json` — tool role
- `adapters/creative/core/context.json` — creative role
- `adapters/code/core/context.json` — code role
- Extract from `nerve_template.py` PLANNER_PROMPT_TOOL/CREATIVE/CODE

#### 1.3 Create core sense prompts
- `adapters/awareness/core/context.json` — from awareness/nerve.py
- `adapters/communication/core/context.json` — from communication/nerve.py
- `adapters/vision/core/context.json` — from sight/nerve.py (image description, detail focus, response format)

#### 1.4 Protect core prompts
- Update `.github/workflows/validate-pr.yml` to block direct edits to `adapters/*/core/`
- Core prompts are system-modified only (via qualification + dream state contribution)

#### 1.5 Update manifest generation
- `scripts/generate_manifest.py` already handles adapters — verify it picks up new structure
- Workflow trigger paths already include `adapters/**` via existing generate-manifest.yml — but verify

### Phase 2: Prompt Loading (sentient-core + sentient-server)

**Goal**: Runtime loads prompts from adapter files instead of hardcoded strings.

#### 2.1 Adapter resolver module
- New: `sentient/brain/adapters.py`
- `resolve_prompt(role, model_name, size_class) -> dict`
  - Tries: exact model → size_class → core
  - Looks in: local `.community/cache/adapters/` first, then fetches from GitHub
  - Returns: `{system_prompt, few_shot_examples, temperature, ...}`
- Cache adapter files locally (same pattern as `community.py` manifest caching)

#### 2.2 Wire into brain routing
- `brain/prompt.py`: `get_system_prompt()` calls `resolve_prompt("brain", current_model, size_class)`
- Falls back to current hardcoded prompt if no adapter found (backward compat)
- Dynamic calibration section still injected at runtime (Redis sense status)

#### 2.3 Wire into nerve template
- `brain/nerve_template.py`: Planner prompts loaded via `resolve_prompt("nerve"|"creative"|"code", ...)`
- Template string still has the runtime code, but PLANNER_PROMPT_* come from adapter

#### 2.4 Wire into senses
- `senses/awareness/nerve.py`: Load base prompt from adapter, overlay personality at runtime
- `senses/communication/nerve.py`: Same pattern

#### 2.5 Model detection at startup
- `brain/config.py` or `inference/config.py`: Expose `get_model_size_class(role) -> str`
- Maps model file/name → size_class (tiny/small/medium/large)
- Used by adapter resolver to pick the right prompt tier

### Phase 3: Wizard Integration (sentient-core create-sentient)

**Goal**: Scaffolding selects the right adapter during project creation.

#### 3.1 Model → adapter lookup
- `wizard/model_selector.py`: After user selects model, query community manifest for matching adapter
- `manifest.adapters[role]` → find by model name or size_class
- Display: "Found optimized prompt for Qwen2.5-7B" or "Using core prompt (no adapter for this model)"

#### 3.2 Scaffold adapter config
- `wizard/scaffold.py`: Write selected adapter info to `sentient.yaml`
  ```yaml
  inference:
    provider: gguf
    models:
      brain: Qwen2.5-Coder-7B
    adapters:
      brain: core          # or "Qwen2.5-Coder-7B" if specific adapter exists
  ```
- Download adapter context.json files to `.community/cache/adapters/`

#### 3.3 Fallback messaging
- If no adapter exists: "No optimized adapter found for {model}. Using core prompt.
  The system will auto-tune in dream state."

### Phase 4: Dream State Evolution (sentient-server)

**Goal**: Running systems evolve prompts and contribute back.

#### 4.1 Prompt tuning in dream state
- `brain/consolidate.py` — new phase: **Prompt Tuning** (after MCP Fanout, before Reconciliation)
- Process:
  1. Load current prompt for each role
  2. Analyze recent routing errors (misroutes from episode history)
  3. Generate improved prompt variant using the brain model
  4. A/B test: run qualification test bank with both prompts
  5. Keep winner, store in cold memory as `adapted_prompt:{role}:{model}`

#### 4.2 Prompt contribution
- After N successful dream cycles with improved scores:
  - Package as adapter: `{model_name}/context.json` + `qualification.json`
  - PR to sentient-community (via GitHub API or manual export)
  - Subject to community review + CI qualification before merge
- This is the "send back" loop — future users with same model get the evolved prompt

#### 4.3 Personality reflection integration
- Dream state personality reflection already evolves voice/traits in cold memory
- Prompt tuning phase should NOT override personality — it tunes routing/planning instructions
- Clear separation: adapter = how to think, personality = how to speak

### Phase 5: Protection & Quality (sentient-community)

#### 5.1 Adapter validation CI
- New workflow: `validate-adapter.yml`
- On PR to `adapters/`:
  - Schema validation (context.json matches adapter_context.schema.json)
  - Prompt length check (core must be < 500 tokens, model-specific < 2000)
  - Required fields present

#### 5.2 Adapter qualification
- Extend `scripts/qualify_adapter.py` (new):
  - Load adapter's system_prompt
  - Run standard routing test bank (10 inputs, check correct action)
  - Score: % correct routing decisions
  - Write qualification.json
- CI runs this on adapter PRs before merge

#### 5.3 Leaderboard
- Manifest already builds leaderboard from adapter scores
- Dashboard could display: "Your brain adapter scores X% — community best is Y%"

## Execution Order

1. **Phase 1** first — create the core prompts in community. This immediately fixes
   the routing bug ("can you tell me a funny joke") by rewriting the brain prompt.
2. **Phase 2** next — wire the loading into sentient-server (test locally).
3. **Phase 3** — wizard integration in sentient-core.
4. **Phase 4** — dream state evolution (longer term).
5. **Phase 5** — CI protection (can start in parallel with Phase 2).

## Immediate Win

Before the full system is built, the brain routing prompt can be fixed inline:
- Remove "ALL conversational exchanges" from awareness scope
- Change priority: existing nerves → senses → synthesize (not senses first)
- Add few-shot examples: "tell me a joke" → invoke joke_nerve, NOT awareness
- Remove contradictory "respond directly" instruction

This fixes the current bug while the adapter system is built.
